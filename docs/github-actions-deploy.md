# GitHub Deploy Guide

This repository is ready for GitHub-based CI/CD using:

- `CI`
- `Deploy Frontend to Vercel`
- `Deploy Backend to Railway`
- `Migrate Neon Database`

Workflow files:
- `C:\laragon\www\HRMS\.github\workflows\ci.yml`
- `C:\laragon\www\HRMS\.github\workflows\deploy-frontend-vercel.yml`
- `C:\laragon\www\HRMS\.github\workflows\deploy-backend-railway.yml`
- `C:\laragon\www\HRMS\.github\workflows\migrate-neon.yml`

## 1. GitHub Repository Setup

1. Create an empty GitHub repository.
2. Keep the default branch as `main`.
3. Do not add GitHub's autogenerated README or `.gitignore` if you want a clean first push from this local repo.
4. After local push, open **Settings > Secrets and variables > Actions** in GitHub.

## Recommended GitHub Secrets

### App runtime
- `JWT_ACCESS_SECRET`
- `JWT_REFRESH_SECRET`

### Vercel
- `VERCEL_TOKEN`
- `VERCEL_ORG_ID`
- `VERCEL_PROJECT_ID`

### Railway
- `RAILWAY_TOKEN`
- `RAILWAY_PROJECT_ID`
- `RAILWAY_ENVIRONMENT`
- `RAILWAY_SERVICE`

### Neon
- `NEON_DATABASE_URL`

## 2. Vercel Setup

### Create the project
1. Log in to Vercel.
2. Create a new project from the GitHub repository.
3. Set the project root to `frontend/`.
4. Confirm the framework is Vite.
5. The repo already includes [C:\laragon\www\HRMS\frontend\vercel.json](C:\laragon\www\HRMS\frontend\vercel.json) so SPA route refreshes rewrite to `index.html`.

### Add environment variables in Vercel
1. Open the Vercel project.
2. Go to **Settings > Environment Variables**.
3. Add:
   - `VITE_API_BASE_URL` = your Railway backend URL + `/api/v1`

## 3. Railway Setup

### Create backend service
1. Log in to Railway.
2. Create a new project.
3. Create an **Empty Service** for the Go backend.
4. Note these values from Railway:
   - project id
   - environment name
   - service name
5. Deploy the `backend/` directory. The repo already includes [C:\laragon\www\HRMS\backend\Dockerfile](C:\laragon\www\HRMS\backend\Dockerfile) so Railway can build the API predictably.

### Runtime environment variables
The GitHub deploy workflow now syncs these Railway variables before each deploy:
- `APP_ENV=production`
- `FRONTEND_ORIGIN=https://hrms-codeid.vercel.app`
- `DATABASE_URL` from `NEON_DATABASE_URL`
- `JWT_ACCESS_SECRET` from GitHub secrets
- `JWT_REFRESH_SECRET` from GitHub secrets

### Recommended Railway setting
- If you also connect the repo directly in Railway, enable **Wait for CI** or disable Railway auto-deploys to avoid duplicate deploys.

## 4. Neon Setup

### Create the production database
1. Log in to Neon.
2. Create a project for production.
3. Copy the primary branch connection string.
4. Use that connection string as `NEON_DATABASE_URL` in GitHub secrets.

### Migration flow
- The workflow `migrate-neon.yml` runs `go run ./cmd/migrate up` against `NEON_DATABASE_URL` on push to `main` when migration files change.
- Keep `seed-admin` and `seed-demo` manual. Do not auto-seed production demo data.

## 5. Recommended First Deploy Order

1. Push code to GitHub.
2. Configure Neon and get `NEON_DATABASE_URL`.
3. Add all required GitHub secrets, including JWT secrets.
4. Run the Neon migration workflow or let it run from a push.
5. Run the Railway backend deploy workflow.
6. Configure Vercel with `VITE_API_BASE_URL` pointing to Railway.
7. Deploy frontend to Vercel.
8. Test login and `/healthz`.

## 6. Recommended Smoke Test After Deploy

1. Open backend `/healthz`.
2. Open frontend URL.
3. Login with admin account.
4. Check employee list loads.
5. Check management scope filter works.
6. Check manager login and leave/attendance page.
7. Check employee self-service attendance and leave.

## 7. Important Notes

- Backend now supports `DATABASE_URL`, which is the easiest path for Railway and Neon.
- Frontend should always use the public Railway backend URL in `VITE_API_BASE_URL`.
- For production, keep JWT secrets unique and strong.
- If CORS errors appear, verify `FRONTEND_ORIGIN` in Railway matches the final Vercel domain.
